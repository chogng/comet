# Smoke reliability migration

## Temporary scope

This migration owns deterministic smoke fixtures, diagnostic waits, and
bounded failure artifacts:

- `test/automation/**` where application/process ownership and condition waits
  are implemented;
- `test/smoke/src/fixtures.ts` and smoke call sites using those fixtures;
- the smoke result reporter and artifact manifest modules under `test/smoke`;
- package scripts needed to run that reporter;
- `.github/workflows/ci.yml` only for failure-artifact upload and retention;
  and
- the fixture, wait, cleanup, and diagnostic rules in
  `test/smoke/README.md`.

It depends on the declaration cutover in
[Smoke test runner](test-runner.migration.md), but does not own Mocha removal or
the CI platform matrix.

## Boundary being replaced

The existing smoke harness owns processes and temporary directories but does
not have a result-aware artifact protocol. A `TestContext.after` hook can stage
diagnostics, but it cannot reliably decide whether the final result passed:
the final outcome may be changed by a later teardown error, timeout, reporter
error, or test-process loss. Failure-only promotion therefore belongs to the
result reporter, not the fixture.

Some flows also use elapsed delays as readiness synchronization. Those waits
can pass or fail according to scheduler timing and do not report the last
state that prevented progress.

## Final project-owned target

### Test-local ownership and waits

Each smoke test owns its application, local server, processes, ports,
temporary root, user-data directory, automation handles, and cancellation
state through one fixture. Cleanup is registered before startup and is
idempotent after partial startup, renderer loss, shared-process exit, timeout,
or assertion failure.

Correctness synchronization waits for a named observable condition with a
bounded deadline and includes the last observed state in its timeout error. A
fixed delay remains only when elapsed time is the explicit input under test;
the assertion still waits for the named outcome.

### Reporter-owned result protocol

The fixture writes diagnostics only to a per-test staging directory. Before
the first fallible startup operation it persists an opaque staging token and
emits one machine-readable `test:diagnostic` record containing that token. The
smoke runner is serial, so the custom `node:test` reporter requires exactly one
active leaf test when it receives the record. It associates the token with
that leaf and derives the canonical test ID from the final event's file,
reconstructed suite path, name, and test number. Zero or multiple active
owners is an infrastructure failure.

The fixture may finalize logs, a screenshot, trace data, process exits, and
metadata during teardown, but it never labels the result passed or failed.
The reporter acts only on the final result event:

- final pass removes that test's staging directory;
- failure, timeout, teardown failure, or cancellation validates and promotes
  the staging directory to the retained failure root; and
- when a test process exits before a final event, the parent runner marks every
  unresolved manifest from that process as failed and promotes its available
  staging data.

No test reads an attachment-like snapshot to create artifacts. Diagnostics
are generated by the smoke fixture only because that test was explicitly
started, and promotion occurs only because its final result failed.

Artifact capture errors are reported in addition to the original failure. On
an otherwise passing test, an invalid, unredacted, missing, or oversized
manifest makes the smoke run fail as a test-infrastructure error.

### Bounded and redacted artifacts

Only synthetic fixture content and redacted diagnostics may be retained. Per
test, the manifest enforces:

- at most 1 MiB of tail logs for each owned process;
- one screenshot no larger than 8 MiB;
- one metadata-only diagnostic JSON file no larger than 64 KiB;
- one Playwright trace no larger than 32 MiB, captured without source files;
  and
- at most 48 MiB for the complete promoted bundle.

Credentials, cookies, authorization headers, secret query values, local user
paths, article or PDF content, chat transcripts, selections, and attachment
payloads are redacted or omitted before staging. The manifest contains file
names, sizes, hashes, redaction status, process roles, exit details, and the
canonical test ID; it does not embed payload content.

Validation happens before promotion. An oversized or disallowed payload is
deleted from the candidate bundle and never uploaded. The reporter retains a
bounded metadata-only rejection record containing the file role, observed
size, hash where safe, and rejection reason, and fails the smoke run.

The reporter consumes the internal staging record instead of printing it. Test
output prints only a retained failure manifest path. CI uploads promoted
failure bundles for seven days and never uploads successful staging data.

## Direct migration steps

1. Make the smoke fixture test-local and register ordered, idempotent cleanup
   before startup. Cover every partial-startup and process-loss edge.
2. Replace correctness sleeps in `test/automation` and smoke call sites with
   named bounded condition waits carrying last-state diagnostics.
3. Add per-test staging and the machine-readable diagnostic manifest record.
   Keep outcome decisions out of fixture hooks.
4. Add the result reporter and parent-runner unresolved-manifest handling.
   Validate canonical ownership before deleting or promoting a directory.
5. Add redaction, per-file caps, total caps, hashes, and deterministic manifest
   serialization. Treat a validation failure as an additional infrastructure
   error.
6. Configure CI to upload only promoted manifests and bundles on failure with
   seven-day retention.
7. Exercise the reliability matrix, then delete this document once every
   completion criterion holds.

## Required conformance cases

- Normal success removes staging and emits no uploadable bundle.
- Assertion failure, timeout, setup failure, teardown failure, cancellation,
  renderer loss, shared-process exit, and parent-observed process loss retain
  the addressed test's available artifacts.
- A final teardown failure promotes a test that had already emitted a passing
  body result.
- One test's result cannot delete or promote another test's staging directory,
  and a diagnostic with zero or multiple active leaf owners fails.
- A missing final event promotes every unresolved manifest for that process.
- Capture failure preserves the original failure and reports the capture
  error; capture failure on a pass fails the run.
- Logs are tail-truncated, secret fixtures are redacted, disallowed content is
  rejected, each individual and total size cap is enforced, and rejected
  payloads never enter the uploaded bundle.
- Manifest serialization is deterministic and contains hashes rather than
  diagnostic payloads.
- Condition waits pass on the named state and report the last state on timeout.
- Late automation completion after timeout cannot mutate a closed fixture.
- Repeated cleanup after partial or complete startup is safe and leaves no
  process, server, port, or temporary root alive.

## Completion and deletion criteria

This migration is complete only when every correctness wait is condition
based, every smoke resource is test-owned and unconditionally cleaned, the
reporter is the sole pass/failure promotion authority, all artifact limits and
redaction rules are mechanically enforced, CI retains only failed bundles for
seven days, and every reliability conformance case passes on Ubuntu and
macOS.

Delete this document in the same change that satisfies these criteria.
